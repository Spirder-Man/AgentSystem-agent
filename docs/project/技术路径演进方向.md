## 二、为什么工具调用必须最先做

### 2.1 依赖关系

```
工具调用架构 ──┬──→ ReAct（推理-行动循环）
              ├──→ CoT （思维链需要真实数据推理）
              ├──→ Reflection（反思需要工具验证事实）
              └──→ 记忆机制（工具执行结果 → 领域事实缓存）
```

工具调用是四大能力的共同基石。工具不动，其他三个范式无法从 demo 升级到企业级，记忆也没有领域数据可记。

### 2.2 当前工具层的致命问题

RAG 知识库管道已能检索 GB30000 / GB15603 / GB50160 全文，但工具根本不走 RAG：

```csharp
// ❌ 当前：15 条硬编码字典
private static readonly Dictionary<string, string> HazardCategories = new()
{
    ["易燃液体"] = "GB 30000.7-2013",
    // 写死在代码里，从未查询知识库
};

// ✅ 目标：工具调用 RAG 检索知识库真实国标文本
//   1. LLM 语义分析 → 确定检索目标
//   2. HybridKnowledgeBaseService.RetrieveChemicalRegulationAsync()
//   3. LLM 基于检索结果生成工具返回
```

| 维度 | 当前 | 目标 |
|------|------|------|
| 工具选择 | 纯关键词匹配 | LLM 语义推理 + 关键词兜底 |
| 工具数据来源 | 硬编码字典（15+5+7=27 条规则） | RAG 检索知识库（数千条国标全文） |
| 工具注册 | `ChemicalComplianceTools.cs` 与 `ToolDefinition` 重复定义 | 统一配置驱动 |
| 工具调用入口 | `AgentDialog.CallTool` 和 `ToolService.CallToolAsync` 两套重复逻辑 | 统一 `ToolService` 调度 |

| 2 | 工具结果注入推理上下文 | 每次工具执行结果写回上下文，LLM 可基于上一轮结果决定下一轮是否需要新工具 |

## 五、不变的设计原则

1. **场景锚定**：所有决策源于化工合规真实痛点，不做空中楼阁
2. **先易后难**：RAG → 工具 → ReAct → 记忆 → CoT/Reflection，每次只改一个维度
3. **非侵入式演进**：通过接口抽象 + DI 注入，新能力装饰旧代码而非重写
4. **配置驱动**：工具定义、模型参数、检索策略全部外部化到 `appsettings.json`
5. **验证先行**：每个阶段完成后必须 `dotnet build` + `dotnet test` 通过
