## 修复详情

### Bug 1 — 重排序 key 对不上

**文件**: `Agent1/ChemicalRAG.cs` — `SearchAsync` 方法

**错误代码**:
```csharp
// priorityLevels key 是 "国标"/"园区规则"/"历史案例"
// 但查找的是 Metadata["Priority"]，值是 "高"/"中"/"低"
r.Metadata["Priority"]   // 读出来 "高"，但字典里没有这个 key
```

**修复**:
```csharp
r.Metadata["RegulationType"]   // 读出来 "国标"，字典里有
// 同时补上 "企业制度" → 500
```

---

### Bug 2 — Clear() 只清一半

**文件**: `Agent1/Services/HybridKnowledgeBaseService.cs` — `Clear` 方法

**错误代码**:
```csharp
public void Clear() { _bm25Service.Clear(); }  // 只清了内存
```

**修复**:
- `IDatabaseService` 新增 `ClearChemicalDocumentsAsync()`
- `DatabaseService` 实现 `DELETE FROM chemical_documents`
- `Clear()` 中调用：`_databaseService.ClearChemicalDocumentsAsync()`

---

### Bug 3 — embedding 失败随机向量污染 PostgreSQL

**文件**: `Agent1/Services/LlmService.cs`, `HybridKnowledgeBaseService.cs`

**错误代码**:
```csharp
// LlmService — Ollama 失败时
catch { return fallback; }  // 768 个随机小数
// HybridKnowledgeBaseService — 直接写库
record.Embedding = embedding;  // 随机数当真实向量
_databaseService.AddChemicalDocumentAsync(record);
```

**修复**:
- `GetEmbeddingAsync` 返回类型改为 `float[]?`，失败返回 `null`
- 调用方检查 `null`，跳过数据库写入

---

### Bug 4 — _partialFiles 重复计数

**文件**: `Agent1/ChemicalRAG.cs` — `ProcessPdfFileAsync`

**错误代码**:
```csharp
// 第 270 行提前判断
if (cleanResult.IsGarbled) _partialFiles++;
// ... ...
// 第 289 行又判断一次
if (cleanResult.IsGarbled) _partialFiles++; else _successFiles++;
```

**修复**: 删除第 270 行的提前判断，只在末尾做一次分类。

---

### Bug 5 — TXT 文件绕过清洗管道

**文件**: `Agent1/ChemicalRAG.cs` — `LoadDirectoryAsync` / `LoadAndSplitFile`

**错误代码**:
```csharp
// TXT 只做了 500 字符切分，没走 TextCleaner + SemanticChunker
var chunks = await LoadAndSplitFile(f, regulationType, priority);
```

**修复**: TXT 也走同一套 `TextCleaner → SemanticChunker` 管道，与 PDF/DOC 路径对齐。

---

### Bug 6 — 降级 BM25 结果被当向量分重复融合

**文件**: `Agent1/Services/HybridKnowledgeBaseService.cs` — `VectorRetrieveAsync`

**错误代码**:
```csharp
catch {
    return await Bm25RetrieveAsync(query, topK);  // 降级返回BM25
}
// 融合阶段：这份BM25结果被当成 vectorScore，和正常 bm25Score 相加
```

**修复**: 降级时返回空列表，让融合代码只用正常 BM25。

---

### Bug 7 — GetChunkKey 哈希碰撞

**文件**: `Agent1/Services/HybridKnowledgeBaseService.cs` — `GetChunkKey`

**错误代码**:
```csharp
chunk.Content.Substring(0, 100).GetHashCode().ToString()
// 前100字符相同 → 哈希相同 → 不同块被当成同一个，后者覆盖前者
```

**修复**: 直接用 `Content` 全文作为去重 key，`null` 时用 `Guid` 兜底。

---

### Bug 8 — BM25/向量数量级不一致导致加权失效

**文件**: `Agent1/Services/HybridKnowledgeBaseService.cs` — `HybridRetrieveAsync`

**错误代码**:
```csharp
hybridScore = bm25Score * 0.4 + vectorScore * 0.6
// bm25Score 典型值 50~150，vectorScore 0~1
// 50*0.4 = 20 完全碾压 0.6*0.6 = 0.36
```

**修复**: BM25 分先除以本批次最大值归一化到 0~1，再与向量分加权。

---

### Bug 9 — AddDocumentsAsync 只写 BM25

**文件**: `Agent1/Services/HybridKnowledgeBaseService.cs`

**错误代码**:
```csharp
return _bm25Service.AddDocumentsAsync(contents);  // 只写内存
```

**修复**: 逐条调用自己的 `AddDocumentAsync`，保证 BM25+向量双写。

---

### Bug 10 — RetrievedChunk.ToString() null 崩溃

**文件**: `Agent1/Services/RetrievedChunk.cs`

**错误代码**:
```csharp
Content.Length > 50  // Content 可能为 null → NRE
```

**修复**: `Content ?? ""` 防御。

---

### Bug 11 — SearchAsync 预览 null 崩溃

**文件**: `Agent1/ChemicalRAG.cs` — `SearchAsync`

**错误代码**:
```csharp
result.Content.Substring(0, ...)  // null → NRE
```

**修复**: `(result.Content ?? "").Substring(...)` 防御。

## 审查经验总结

| 审查维度 | 问自己 |
|----------|--------|
| Key 一致性 | 存的 key 和查字典的 key 是同一组吗？ |
| 子系统完整性 | 类有几个存储？清理了几个？ |
| 降级副作用 | catch 返回的兜底值会进数据库吗？ |
| 计数器守恒 | 事件 1 次，计数器 +1 还是 +N？ |
| 路径对称性 | 两条分支步骤数一样吗？ |
| 量纲一致性 | 两个数要加权，数量级差多少倍？ |
