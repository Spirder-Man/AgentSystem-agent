## 一、修复背景

### 1.1 项目定位

本项目是面向**化工园区危化品合规审核**的企业级 AI Agent 系统，基于 .NET 8 + Semantic Kernel + Ollama 构建，核心能力包括：

- RAG 检索增强生成（BM25 + pgvector 混合检索）
- CoT 思维链推理（流式/标准双模式）
- ReAct 交互式推理（Thought → Action → Observation 循环）
- Reflection 自我反思纠错
- 化工合规自查（国标/园区规则/历史案例三级知识库）

### 1.2 核心问题：功能错位

项目最初的模板代码来自**工业制造场景**（机床主轴温度监控），后来转向化工合规领域，但大量代码**残留了工业工具的硬编码引用**。典型症状：

| 场景 | 用户提问 | 系统实际调用 | 问题 |
|------|----------|-------------|------|
| 储存合规 | "氧化剂和易燃液体能一起存吗" | `GetSpindleTemperature()` | 返回随机主轴温度 190-220℃，与化工无关 |
| 危险分类 | "苯属于什么危险类别" | `GetTemperatureThreshold()` | 返回"主轴安全温度阈值 ≤ 180℃" |
| 安全距离 | "储罐之间的安全距离是多少" | `GetSpindleTemperature()` | 返回机床主轴温度，牛头不对马嘴 |

**根本原因**：RAG 模块的 Prompt 已经切换为化工合规专家角色，但底层工具调用链路仍然绑定 `IndustrialTools`（机床温度/主轴阈值），形成了"化工问题 + 工业工具"的功能错位。

### 2.2 新增文件：ChemicalComplianceTools.cs

#### 设计思路

`ChemicalComplianceTools` 是面向化工合规场景的全新工具集，替代工业制造场景的 `IndustrialTools`。核心设计原则：

1. **领域专一**：每个工具对应一项化工合规审查能力
2. **法规对齐**：工具返回结果直接引用具体国标编号
3. **Semantic Kernel 兼容**：使用 `[KernelFunction]` 属性标注，支持 SK 自动发现和注入
4. **知识库兜底**：无法精确匹配时，引导用户查阅 `knowledgebase/` 目录下的标准全文

#### 完整代码实现

```csharp

#### 五个工具方法详解

**工具 1：CheckHazardCategory(string substanceName)**

- **功能**：查询危化品属于哪个危险类别，返回对应的 GB 30000 系列标准号
- **参数演变**：`substanceName` — 从用户自然语言中提取的危化品名称（由后续 `ExtractSubstance()` 预处理）
- **匹配逻辑**：双向模糊匹配（`substanceName.Contains(kvp.Key) || kvp.Key.Contains(substanceName)`），兼顾"苯"匹配"易燃液体"的场景
- **兜底策略**：未匹配时引导查阅 `knowledgebase/国标/` 目录下的完整标准文件

```
修复前（IndustrialTools.GetSpindleTemperature）:
  → 返回随机主轴温度值（190-220℃），化工问题完全无法使用

修复后（ChemicalComplianceTools.CheckHazardCategory）:
  → 输入"苯" → 输出"属于「易燃液体」类别，适用标准：GB 30000.7-2013"
```

**工具 2：CheckStorageCompatibility(string substanceA, string substanceB)**

- **功能**：检查两种危化品是否可以同库储存，依据 GB15603-1995 第4.2.2条
- **参数演变**：两个危化品名称（由 `ExtractTwoSubstances()` 从"氧化剂和易燃液体能一起存吗"中拆分提取）
- **判断逻辑**：
  1. 遍历 `StorageIncompatibilities` 字典
  2. 检查 `substanceA` 是否匹配某禁忌类别、`substanceB` 是否在该类别的禁忌列表中
  3. 匹配成功 → 返回 `⚠️ 禁用` + GB15603 条款引用
  4. 匹配失败 → 返回 `✅ 未发现直接冲突` + 建议核实

```
修复前（IndustrialTools.GetTemperatureThreshold）:
  → 返回"主轴安全温度阈值：≤ 180℃"

修复后（ChemicalComplianceTools.CheckStorageCompatibility）:
  → 输入("氧化剂", "易燃液体") → 输出"⚠️ 禁用：氧化剂类不可与之同库贮存。依据：GB15603-1995 第4.2.2条"
```

**工具 3：GetSafetyDistance(string facilityType)**

- **功能**：查询设施间的安全间距要求
- **参数演变**：设施类型字符串（由 `ExtractFacilityType()` 从"储罐之间的安全距离是多少"中推断为"储罐-储罐"）
- **两级匹配**：
  1. 精确匹配 `SafetyDistances.TryGetValue(key, out distance)`
  2. 模糊匹配 `SafetyDistances.Keys.Where(k => k.Contains(key) || key.Contains(k))`
- **兜底**：引导查阅 GB50160 和 GB50016 全文

**工具 4-5：GetCurrentTime() / Calculate(string expression)**

- 保留自 `IndustrialTools`，属于通用工具，化工场景同样适用
- `Calculate` 使用 `System.Data.DataTable().Compute()` 安全计算数学表达式

// ── 修复后（配置驱动匹配）──
public async Task<ToolPlan> AnalyzeAndPlanToolsAsync(string userInput, string history)
{
    var lowerInput = userInput.ToLower();
    foreach (var tool in _toolDefinitions)
    {
        // 遍历 ToolDefinition.KeywordTriggers，任一词匹配即触发
        if (tool.KeywordTriggers.Any(kw => lowerInput.Contains(kw.ToLower())))
        {
            plan.NeedsTools = true;
            plan.ToolNames.Add(tool.Name);
            return plan;  // 首次匹配即返回
        }
    }
    plan.NeedsTools = false;
    return plan;
}
```

## 四、Bug 修复

### 4.1 修复 1：ParseToolCalls 误匹配问题（🔴 严重）

#### 问题场景

用户提问："储罐之间的安全距离是多少"

LLM 在 Step1 Thought 中分析道：
```
CheckHazardCategory：用于查询危化品危险类别...（不适用）
CheckStorageCompatibility：检查两种危化品是否可以同库储存...（不适用）
GetSafetyDistance：查询设施间安全间距要求...这个正好适用
因此，只需要调用 GetSafetyDistance 这个工具。
```

旧版 `ParseToolCalls` 使用 **全文 Contains 匹配**：
```csharp
// ── 修复前：全文本扫描 ──
foreach (var toolName in toolNames)
{
    if (modelOutput.Contains(toolName))  // 危险！LLM 思考列举了所有工具名
    {
        tools.Add(toolName);
    }
}
// 结果：CheckHazardCategory, CheckStorageCompatibility, GetSafetyDistance, GetCurrentTime, Calculate
//       全部 5 个工具被误匹配！
```

**后果链**：
```
5 个工具全被调用 → 参数全是完整问句 → CheckHazardCategory("储罐之间的安全距离是多少") 无法匹配
→ GetSafetyDistance("储罐之间的安全距离是多少") 精确匹配失败
→ 最终 LLM 因缺乏真实数据而编造答案
```

#### 修复方案：标记行限制匹配

```csharp
// ── 修复后：只在 需要调用的工具： 标记行中匹配 ──
string marker = "需要调用的工具：";
int markerIndex = modelOutput.IndexOf(marker, StringComparison.OrdinalIgnoreCase);

if (markerIndex >= 0)
{
    // 提取标记行内容（只取第一行）
    string afterMarker = modelOutput.Substring(markerIndex + marker.Length);
    int newlineIndex = afterMarker.IndexOf('\n');
    string toolLine = newlineIndex >= 0 ? afterMarker.Substring(0, newlineIndex) : afterMarker;
    
    // 检查是否明确为"无"
    string cleanLine = toolLine.Trim().Replace("。", "").Replace(".", "").Trim();
    if (cleanLine == "无" || string.IsNullOrEmpty(cleanLine))
        return new string[0];
    
    // 只在标记行内匹配工具名
    foreach (var toolName in toolNames)
        if (toolLine.Contains(toolName, StringComparison.OrdinalIgnoreCase))
            tools.Add(toolName);
}
else
{
    // 没找到标记行 → 保守返回空，不让无关工具被误调用
    return new string[0];
}
```

**同时强化 Thought Prompt 格式约束**，在 prompt 末尾增加：
```
请分析是否需要调用工具。注意：你的思考可以自由发挥，但最后一行必须是以下格式（二选一）：
需要调用的工具：工具名1,工具名2
需要调用的工具：无
```

这样 LLM 会在输出末尾产生明确的标记行，ParseToolCalls 可以精确捕获。

### 4.2 修复 2：智能参数提取（🟡 中等）

#### 问题场景

旧版 `CallTool` 直接把**整个用户问句**传给工具方法：

```csharp
// ── 修复前 ──
"CheckHazardCategory" => tools.CheckHazardCategory(userInput),
// userInput = "氧化剂和易燃液体能一起存吗"
// 工具内部: substanceName.Contains("易燃液体") → 意外匹配
// 期望匹配的是"氧化剂"但返回了"易燃液体"类别

"GetSafetyDistance" => tools.GetSafetyDistance(userInput),
// userInput = "储罐之间的安全距离是多少"
// 工具内部: SafetyDistances.TryGetValue("储罐之间的安全距离是多少", out _) → 失败
```

#### 修复方案：按工具类型智能提取参数

新增 5 个参数提取辅助方法：

**1. ExtractSubstance(string userInput)** — 提取单个危化品名称

```csharp
private string ExtractSubstance(string userInput)
{
    return userInput
        .Replace("属于什么危险类别", "").Replace("是什么类别", "")
        .Replace("的危化品", "").Replace("是什么", "")
        .Replace("属于什么", "").Replace("什么", "")
        .Replace("是多少", "").Replace("能不能", "")
        .Replace("吗", "").Replace("？", "").Replace("?", "").Trim();
}
// "苯属于什么危险类别" → "苯"
// "氧化剂和易燃液体能一起存吗" → "氧化剂和易燃液体能一起存"
```

**2. ExtractTwoSubstances(string userInput)** — 提取两个危化品名称

```csharp
private (string a, string b) ExtractTwoSubstances(string userInput)
{
    var separators = new[] { "和", "与", "跟", "、" };
    foreach (var sep in separators)
    {
        if (userInput.Contains(sep))
        {
            var parts = userInput.Split(new[] { sep }, 2, StringSplitOptions.RemoveEmptyEntries);
            if (parts.Length >= 2)
            {
                string a = CleanSubstance(parts[0]);  // "氧化剂"
                string b = CleanSubstance(parts[1]);  // "易燃液体能一起存吗" → "易燃液体"
                if (!string.IsNullOrWhiteSpace(a) && !string.IsNullOrWhiteSpace(b))
                    return (a, b);
            }
        }
    }
    return (ExtractSubstance(userInput), "");  // 兜底
}
```

**3. CleanSubstance(string text)** — 清理物质名中的问句残留

```csharp
private string CleanSubstance(string text)
{
    return text.Replace("能一起存", "").Replace("可以一起", "").Replace("能不能", "")
               .Replace("可以", "").Replace("吗", "").Replace("？", "").Replace("?", "")
               .Replace("怎么", "").Replace("如何", "").Trim();
}
```

**4. ExtractFacilityType(string userInput)** — 推断设施类型

```csharp
private string ExtractFacilityType(string userInput)
{
    if (userInput.Contains("液化烃")) return "液化烃储罐-储罐";
    if (userInput.Contains("仓库") && userInput.Contains("明火")) return "甲类仓库-明火点";
    if (userInput.Contains("仓库") && userInput.Contains("建筑")) return "甲类仓库-建筑";
    if (userInput.Contains("储罐") && userInput.Contains("建筑")) return "储罐-建筑";
    if (userInput.Contains("储罐") && userInput.Contains("消防")) return "储罐-消防通道";
    if (userInput.Contains("储罐") && userInput.Contains("边界")) return "储罐-厂区边界";
    if (userInput.Contains("储罐")) return "储罐-储罐";  // 默认
    return userInput;  // 兜底
}
```

**5. CallStorageCheck(ChemicalComplianceTools, string)** — 包装方法

```csharp
// switch 表达式不支持内联变量声明 (var a,b = ...)，故抽离为独立方法
private string CallStorageCheck(ChemicalComplianceTools tools, string userInput)
{
    var (a, b) = ExtractTwoSubstances(userInput);
    return tools.CheckStorageCompatibility(a, b);
}
```

#### 重构后的 CallTool

```csharp
private string CallTool(ChemicalComplianceTools tools, string toolName, string userInput)
{
    return toolName switch
    {
        "CheckHazardCategory" => tools.CheckHazardCategory(ExtractSubstance(userInput)),
        "CheckStorageCompatibility" => CallStorageCheck(tools, userInput),
        "GetSafetyDistance" => tools.GetSafetyDistance(ExtractFacilityType(userInput)),
        "GetCurrentTime" => tools.GetCurrentTime(),
        "Calculate" => tools.Calculate(userInput),
        _ => $"未知工具: {toolName}"
    };
}
```

#### 修复前后对比

| 用户输入 | 工具 | 修复前参数 | 修复后参数 |
|----------|------|-----------|-----------|
| "氧化剂和易燃液体能一起存吗" | CheckStorageCompatibility | ("氧化剂和易燃液体能一起存吗", "") → 匹配失败 | ("氧化剂", "易燃液体") → ✅ 命中禁忌表 |
| "苯属于什么危险类别" | CheckHazardCategory | "苯属于什么危险类别" → 匹配失败 | "苯" → ✅ 匹配"易燃液体" |
| "储罐之间的安全距离是多少" | GetSafetyDistance | "储罐之间的安全距离是多少" → 精确匹配失败 | "储罐-储罐" → ✅ 返回 15 米 |

### 4.3 修复 3：校验 Prompt 格式约束（🟡 中等）

#### 问题场景

Step5 校验阶段，LLM 收到的 prompt 包含 `{initialConclusion}`（初步回答的全部内容），但没有明确的格式约束。LLM 倾向于**重新复述一遍初步回答**，而非执行校验。

```
实际表现：
  → LLM输出包含"好，我现在要解决用户的问题..."开头的完整分析
  → 校验逻辑被稀释，失去了纠错意义
  → 产生大量冗余 token，增加了后续 Step6 的上下文负担
```

#### 修复方案：三重格式约束

```csharp
string validatePrompt = $@"【任务】校验下面【需要校验的回答】是否基于真实资料，禁止复述或重新分析！

重要：请严格区分三类信息：
1. 用户的假设问题（用户说的）
2. 真实知识库资料（系统提供的）
3. 真实工具数据（系统实际调用工具返回的）

...（检查要点）...

输出要求（严格遵守）：
- 不要复述初步回答的内容        ← 约束1：禁用自我复制
- 不要重新分析问题              ← 约束2：禁用二次推理
- 只输出一行：通过/不通过 + 一句简短原因  ← 约束3：强制单行输出
- 示例：通过，所有数据均来自工具调用结果
- 示例：不通过，回答中引用了知识库中不存在的温度数值";
```

**三个约束的作用机制**：

| 约束 | 解决的问题 | 实现原理 |
|------|-----------|----------|
| "不要复述初步回答的内容" | LLM 自我复制（输出 = 输入） | 通过显式否定指令打断复制倾向 |
| "不要重新分析问题" | LLM 把校验当成了第二轮推理 | 限定角色为"校验员"而非"分析师" |
| "只输出一行 + 示例" | LLM 产生冗长的解释性输出 | 格式示例是 LLM 最有效的约束方式 |
