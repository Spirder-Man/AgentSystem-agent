## V0.1 - 四个基础范式

### 设计目标

建立一套通用的工业AI对话流程框架，确保输出的严谨性和可控性。

***

### 基础范式总览

| 范式  | 英文            | 说明         |
| --- | ------------- | ---------- |
| 1️⃣ | **ReAct**     | 推理+行动+观察循环 |
| 2️⃣ | **ToC**       | 内容大纲+结构化输出 |
| 3️⃣ | **反思纠错**      | 自我检查+修正输出  |
| 4️⃣ | **CoT (思维链)** | 逐步思考过程     |

***

#### 1️⃣ ReAct - 推理与行动范式

**设计思路：**

```
思考 → 行动 → 观察 → 再思考 → ...
```

**V0.1实现方式：**

```csharp
阶段1 - 分析与规划：判断是否需要调用工具
阶段2 - 执行工具调用
```

**特点：**

- ✓ 透明的工具调用过程
- ✓ 工具结果可验证
- ✗ 流程稍显冗长

***

#### 2️⃣ ToC - 内容大纲范式

**设计思路：**
先确定输出的结构，再填充内容，确保输出完整且符合逻辑。

**V0.1实现方式：**

```
【内容大纲】
一、问题分析
   1.1 用户问题
二、工具调用结果
三、诊断结论
   3.1 问题判断
   3.2 故障原因
   3.3 整改建议
```

**特点：**

- ✓ 结构清晰，不易遗漏
- ✓ 符合工业报告习惯
- ✗ 每次都生成ToC，有点耗时

***

#### 3️⃣ 反思纠错 - 自我检查范式

**设计思路：**
输出后进行自检，发现问题再修正，避免幻觉和错误。

**V0.1反思维度：**

| 维度    | 检查内容         |
| ----- | ------------ |
| 数据真实性 | 是否编造，是否有工具支持 |
| 结论严谨性 | 是否符合阈值/规则    |
| 建议落地性 | 是否具体可执行      |
| 结构完整性 | 是否符合ToC要求    |

**特点：**

- ✓ 输出质量高，错误少
- ✓ 减少幻觉
- ✗ 增加一个调用轮次，耗时变长

***

#### 4️⃣ CoT (思维链) - 思考过程范式

**设计思路：**
让模型展示思考的每个步骤，提高可解释性和可靠性。

**V0.1实现方式：**

- 流式输出思考过程
- 但DeepSeek会输出 `<think>` 标签

**特点：**

- ✓ 过程透明，可调试
- ✓ 推理更可靠
- ✗ 展示给用户显得很啰嗦

***

***

### V0.1 完整工作流

```
┌─────────────────────────────────────────┐
│ 用户输入                                   │
└─────────────────────────────────────────┘
                 ↓

                 ↓

│  → 按照ToC结构填充内容                     │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│ 阶段5: 自我反思与纠错 (反思)               │

└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│ 阶段6: 生成最终结论                       │
│  → 基于反思修正问题                       │
│  → 输出最终答案                           │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│ 用户输出（含所有中间过程）                 │
└─────────────────────────────────────────┘
```

## V1.0 - 优化设计方案

### 设计目标

在保持V0.1严谨性的前提下，大幅提升用户体验：

1. **记忆真正工作** - 主动提取并记住关键信息
2. **双模式切换** - 简洁/专业模式可选
3. **流程优化** - 内部推理隐藏，按需查看

***

### 设计原则

| 原则          | 说明               |
| ----------- | ---------------- |
| 🎯 **用户优先** | 默认简洁，想看详情再展开     |
| 🧠 **智能记忆** | 主动提取，而不是硬塞历史     |
| 🔀 **模式分离** | 通用对话 vs 工业诊断分开走  |
| ⚡ **性能优先**  | 能不调工具就不调，能从记忆找就找 |

### 💻 核心代码结构

让我给你看**重新设计的SmartDialogSystem**核心：

```csharp
public class SmartDialogSystem
{
    // ========== 1. 记忆存储 ==========
    private readonly SessionContext _session;
    private readonly UserProfile _userProfile; // 新增：用户档案
    private readonly List<ConversationTurn> _history; // 结构化历史

    // ========== 2. 运行模式 ==========
    public bool IsIndustrialMode { get; set; } // 模式开关

    public async Task<string> ProcessInputAsync(string userInput)
    {
        // ========== 3. 先从记忆里查找 ==========
        var answer = TryAnswerFromMemory(userInput);
        if (!string.IsNullOrEmpty(answer)) 
        {
            return answer; // 不需要任何工具！直接回答！
        }
        
        // ========== 4. 根据模式选择流程 ==========
        if (IsIndustrialMode)
        {
            return await ProcessIndustrialModeAsync(userInput); // 严谨模式
        }
        else
        {
            return await ProcessSimpleModeAsync(userInput); // 快速模式
        }
    }

    // ========== 5. 关键：记忆检索 ==========
    private string TryAnswerFromMemory(string userInput)
    {
        // 例子：用户问"刚才的温度是多少？"
        if (userInput.Contains("温度") && userInput.Contains("刚才"))
        {
            var lastTemperature = _history
                .Where(h => h.KeyFacts.ContainsKey("温度"))
                .LastOrDefault();
                
            if (lastTemperature != null)
            {
                return $"你好！刚才检查的温度是 {lastTemperature.KeyFacts["温度"]}。";
            }
        }
        
        // 例子：用户问"我叫什么"
        if (userInput.Contains("我叫") || userInput.Contains("名字"))
        {
            if (!string.IsNullOrEmpty(_userProfile.UserName))
            {
                return $"你叫 {_userProfile.UserName}，是 {_userProfile.JobTitle}。";
            }
        }
        
        return null; // 从记忆里找不到答案
    }
}
```

***

- 日常问答
- 简单问题
- 不需要严格验证的情况

### V1.0 记忆检索机制

**核心思路：**
每次用户输入，先看**能不能从记忆里回答**，能就**不经过推理/工具调用**，快速响应！

***

#### 记忆检索流程

```
用户输入
   ↓
┌─────────────────────────────────────┐
│  记忆检索器 (Memory Retriever)       │
│  ┌─────────────────────────────────┐ │
│  │ 1. 关键词检测                  │ │
│  │    刚才？ 刚才的？？→ 查历史    │ │
│  │    我叫？ 名字？  → 查用户档案 │ │
│  │    温度多少？  → 查关键事实    │ │
│  └─────────────────────────────────┘ │
│  ┌─────────────────────────────────┐ │
│  │ 2. 相似度匹配 (可选)           │ │
│  │    向量检索相似问题            │ │
│  └─────────────────────────────────┘ │
└─────────────────────────────────────┘
   ↓
有现成答案吗？
   → 有 → 直接回答！✅
   → 没有 → 继续推理流程 ⏳
```

***

#### 检索规则示例

| 问题模式      | 回答来源  | 例子           |
| --------- | ----- | ------------ |
| "刚才..."   | 历史记录  | "刚才的温度是多少？"  |
| "我叫..."   | 用户档案  | "我叫什么名字？"    |
| "温度...阈值" | 关键事实库 | "安全阈值是多少来着？" |

## 版本对比

| 对比项             | **V0.1**    | **V1.0**      |
| --------------- | ----------- | ------------- |
| **输出方式**        | 6阶段全透明给用户看  | 默认简洁，点击看详情    |
| **记忆机制**        | Prompt里硬塞历史 | 结构化存储+主动检索    |
| **工具调用**        | 啥问题都先调工具    | 优先从记忆找，必要才调   |
| **模式**          | 工业和通用都一样    | 双模式可选 (快速/严谨) |
| **`<think>`处理** | 偶尔还有残留显示    | 完整过滤，内部隐藏     |
| **回答速度**        | 较慢（6次调用）    | 快（有时直接从记忆回答）  |
| **用户体验**        | 流程冗长，需要等待   | 即时响应，想看就看     |
| **严谨性**         | ✅ 很高，完整流程   | ✅ 工业模式保持严谨    |
