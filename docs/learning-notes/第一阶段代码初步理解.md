## 二、详细代码结构

### 2.1 IInferenceModule.cs - 统一接口

```csharp
public interface IInferenceModule
{
    string Name { get; }           // 模块名称
    string Description { get; }    // 模块描述
    Task RunAsync();               // 异步运行模块
}
```

### 2.2 模块实现的统一结构

所有 `Modules` 文件夹下的模块都遵循完全一致的结构：

```csharp
public class XxxModule : IInferenceModule
{
    // 1. 模块名称
    public string Name => "模块名称";
    
    // 2. 模块描述
    public string Description => "模块描述";
    
    // 3. 持有对应的范式类
    private readonly 范式类 _xxx;
    
    // 4. 构造函数注入服务并初始化范式类
    public XxxModule(ILlmService llmService, ISessionService sessionService)
    {
        _xxx = new 范式类(llmService, sessionService);
    }
    
    // 5. 实现 RunAsync() 调用范式的异步方法
    public async Task RunAsync()
    {
        await _xxx.范式方法();
    }
}
```

## 三、完整调用链路

### 3.1 从用户选择到执行

```
Program.cs (用户选择菜单)
    ↓
ModuleDispatcher.ExecuteModuleAsync(ModuleType)
    ├─→ 检查缓存字典是否已有该模块
    ├─→ 若没有，调用 ModuleFactory.CreateModule()
    ├─→ 将新模块存入缓存
    ├─→ 输出模块信息
    └─→ 调用 module.RunAsync()
        ↓
    CoTSolidModule.RunAsync() (或其他模块)
        ↓
    CoT.RunCoT() (或对应范式方法)
        ├─→ Console.WriteLine() 输出提示信息
        ├─→ while (true) 循环等待用户输入
        ├─→ 检查 "exit"/"quit"/"退出" 退出条件
        ├─→ _sessionService.AddDialogTurn(SessionId, "User", userInput)
        ├─→ _sessionService.GetFormattedHistory(SessionId, 10)
        ├─→ 构建 Prompt
        ├─→ _llmService.InvokeStreamWithRetryAsync() 调用 LLM
        └─→ _sessionService.AddDialogTurn(SessionId, "Assistant", answer)
```

### 3.2 CoT.RunCoTL() 方法详解（以流式输出为例）

1. **输出提示信息**：显示会话已创建、Session ID、退出命令提示
2. **进入循环**：`while (true)` 持续等待用户输入
3. **读取用户输入**：`Console.ReadLine()`
4. **退出判断**：检查是否为 "exit"/"quit"/"退出"
5. **保存用户输入**：通过 `SessionId` 作为键值存入会话服务
6. **获取历史记录**：最多获取最近10轮对话
7. **构建Prompt**：角色+历史+当前问题+要求
8. **调用LLM**：`_llmService.InvokeStreamAsync()` 流式输出
9. **保存回复**：将LLM回复也存入会话历史
10. **显示完成提示**

## 四、设计模式体现

### 4.1 工厂模式
- `ModuleFactory` 根据 `ModuleType` 枚举创建不同的模块实例
- 客户端不需要知道具体的创建逻辑

### 4.2 策略模式
- 每个 `IInferenceModule` 实现都是一种不同的推理策略
- 可以在运行时动态切换不同的推理策略

### 4.3 依赖注入
- 所有服务（`ILlmService`、`ISessionService` 等）通过构造函数注入
- 避免直接 `new`，提高可测试性和可维护性

### 4.4 缓存机制
- `ModuleDispatcher` 维护一个字典缓存已创建的模块
- 避免重复创建，提高性能

### 4.5 接口隔离
- `IInferenceModule` 定义了最小化的接口
- 各模块只需要关注自己的实现

---

## 五、架构优势

1. **扩展性强**：新增推理模块只需：
   - 实现 `IInferenceModule` 接口
   - 在 `ModuleFactory` 中注册
   - 在 `ModuleType` 枚举中添加

2. **解耦**：
   - 调度器不依赖具体模块实现
   - 各模块之间相互独立
   - 服务层通过接口抽象

3. **可维护**：
   - 统一的接口使得代码结构清晰
   - 职责分离，易于定位问题

4. **符合项目规范**：
   - 严格遵循依赖注入规则
   - 不直接 `new` 服务
   - 通过服务层调用 LLM

### 6.2 RAG.cs 真实执行流程（无美化）

```
1. 启动 → 硬编码读取固定路径两个文件
2. 用户提问 → 暴力拆分文字，提取固定关键词
3. 关键词去文件里数出现次数 → 挑3条最相关的
4. 硬编码写死工具列表 → 让LLM假装思考
5. 不管LLM返回什么 → 强制调用两个固定工具
6. 把「历史+检索内容+工具结果」拼字符串 → 丢给LLM输出
7. 反思、最终结论 → 全是字符串拼接+固定格式
```

**本质：固定流程的字符串拼接工具，不是智能RAG。**

### 6.4 三个致命逻辑硬伤（纯实现级）

#### 硬伤1：决策链完全断裂 - LLM无任何决策权
- 工具调用写死
- 检索关键词写死
- 兜底逻辑写死

- 只能识别固定几个词，稍微换个说法就失效
- 文件路径硬编码，无任何工程化可言

#### 硬伤3：冗余废代码 - ParseToolCalls 完全无用
- 写了解析工具的逻辑
- 又用硬编码覆盖结果
- 代码冗余，逻辑自相矛盾

---

### 7.2 工业级标准 vs 玩具级代码对比

#### 你的正确逻辑（工业级标准）：
1. **系统只给LLM提供真实素材**（知识库、工具数据、历史对话）
2. **LLM自主分析、自主思考、自主决策**（该用什么工具、该怎么回答、有没有问题）
3. **代码只做一件事：校验LLM的输出是否符合真实数据**（反思/纠错/把关）
4. **最终输出 = LLM独立思考的结果**，不是代码拼好的模板

#### 这段代码的错误逻辑（玩具级）：
1. **代码把所有思考步骤写死**（先检索、再调用固定工具、再拼接固定Prompt）
2. **代码把所有话术规则写死**（必须参考案例、必须分析温度、必须给方案）
3. **LLM只能照着代码拼好的稿子念**，没有任何自主判断
4. **最终输出 = 代码模板 + LLM润色**

| **4. 工具调用设计** | 工具全部注册制、配置化，不在代码里写死 switch、写死工具名<br>由 LLM 自主输出需要调用哪些工具，工程层只做解析、路由、执行<br>绝不人工兜底强制覆盖工具列表<br>解析失败 → 交由 LLM 重新生成工具调用 / 进入纯文本回答模式<br>工具参数、工具描述配置化注入，不硬编码写在 Prompt 里 | ParseToolCalls 写了白写，最后还是硬编码强制调用固定工具，完全冗余 | ✅ **完全一致** |
| **5. 幻觉治理+反思纠错** | 第一轮：给素材 → LLM 自主生成原生初步结论<br>工程层带入真实工具数据、知识库原文<br>第二轮：反思校验——让 LLM 自查：是否编造数据、是否超出知识库范围、是否违背阈值规则、是否答非所问<br>第三轮：基于反思结果修正输出<br>工程层最后再加一道规则拦截：明显编造直接拦截、提示信息不足<br>关键：反思纠错不是拼接固定 Prompt 模板，是基于 LLM 原生回答 + 真实原始数据做对比校验 | 先大模型自己思考出解答，再去判断可靠性、再反思纠错，而不是全程字符拼接套模板 | ✅ **完全一致** |

### 8.3 你的专属工业级方法论（可复用、可落地）

**（完全从你踩坑感悟 + 业界架构师标准做法提炼出来，以后你写任何 Agent、RAG、范式都按这套守）**

#### 1. LLM 权责原则
工程层只给原料和规则，业务推理、决策、工具选择、分析思考全权交给 LLM，代码绝不越俎代庖。

#### 2. Prompt 设计原则
禁止超长硬编码业务模板，Prompt 只定角色 + 基础约束 + 输出格式，不干涉 LLM 业务思考逻辑。

#### 3. RAG 检索原则
检索只负责召回相关素材，不做业务解读、不强行兜底、不硬编码关键词和路径，语义优先于暴力关键词匹配。

#### 4. 工具调用原则
工具注册化、配置化，由 LLM 自主决定调用项，工程层只解析执行，绝不强行兜底固定工具。

#### 5. 幻觉与反思纠错原则
固定流程：LLM 原生作答 → 带入真实原始数据 → 反思自查 → 修正输出 → 工程层规则兜底校验。拒绝套模板式反思拼接。

#### 6. 代码与 LLM 边界原则
字符串拼接只拼动态数据，绝不拼接业务逻辑、思考步骤、固定诊断话术。
