### 2.2 KnowledgeBaseService（检索算法核心类）

|字段|类型|技术原理 & 设计价值|
|---|---|---|
|`_documents`|`List<Document>`|内存索引：将所有文档加载到内存，避免每次检索都读文件，提升BM25计算速度；耦合度「中」（仅内部依赖Document）。|

## 三、核心技术流程的原理拆解

系统的核心执行流程包括「知识库加载」「中文分词」「两阶段检索」，每个环节均针对化工合规场景做了定制化设计。

### 3.1 知识库加载（LoadKnowledgeBaseAsync）

**核心职责**：将文件系统中的合规文档加载到内存，并预处理为Document对象。

#### 技术原理：

1. **业务优先级加载**：按「国标→园区规则→历史案例」顺序加载，符合化工合规审核的业务逻辑（国标是最高优先级的合规依据）；

2. **文件分块处理**：调用`SplitTextIntoChunks(text, 500)`将长文档切分为500字符的小块——解决「长文档检索精度低」的问题（小块文档的关键词匹配更精准，也符合LLM上下文窗口的长度限制）；

3. **中文适配**：使用`Encoding.UTF8`读取文件，避免中文乱码（化工文档大量包含中文，是本地化适配的关键）；

4. **容错设计**：单个文件加载失败（try-catch）不影响整体流程，保证知识库加载的鲁棒性；

5. **索引构建**：每个分块生成Document对象后，调用`KnowledgeBaseService.AddDocumentAsync`，内部自动构建「倒排索引、词频统计、文档长度」等核心数据结构。

### 3.2 中文分词（Tokenize）

**核心问题解决**：中文无空格分隔，若将整句作为单个Token，无法匹配用户的关键词（比如用户搜“储罐”，整句Token“危化品储罐安全距离”无法命中）。

#### 技术原理：NGram多粒度分词

NGram是将文本切分为「n个连续字符」的组合，文档中采用多粒度策略：

```Plain Text

示例："危化品储罐安全距离"
- 完整词：保留整句（匹配长关键词）
- 2-gram："危化","化品","品储","储罐"...（匹配双字关键词）
- 单字："危","化","品"...（匹配单字关键词）
```

- **设计策略**：召回优先（宁可多切分，也不遗漏潜在匹配），覆盖用户不同的查询粒度（比如用户搜“安全距离”“储罐”“罐”都能命中）；

- **性能优化**：分词结果缓存到Document的`Tokens`字段，避免重复计算。

### 3.3 两阶段检索（SearchAsync）

检索流程分为「BM25粗召回」和「业务优先级重排序」，核心是「算法召回足够多候选 + 业务规则筛选高优先级结果」。

#### 阶段1：BM25粗召回

**BM25算法核心公式**（简化版）：

```Plain Text

BM25(文档, 查询) = Σ [IDF(关键词) × (TF(关键词,文档) × (K1 + 1)) / (TF(关键词,文档) + K1 × (1 - B + B × 文档长度 / 平均文档长度))]
```

- IDF（逆文档频率）：关键词在越少文档中出现，权重越高（比如“危化品储罐”比“安全”更具辨识度）；

- TF（词频）：关键词在文档中出现次数越多，权重越高（但受K1/B限制，避免过度加权）；

**执行逻辑**：

1. 对用户查询做NGram分词；

2. 调用`RetrieveAsync(query, topK*3)`：基于BM25计算所有文档的得分，返回前`topK*3`个候选（比如topK=5则返回15个）—— 留足候选空间，避免业务重排序后无高优先级结果；

3. 底层依赖`_termDocFreq`倒排索引快速匹配关键词，无需遍历所有文档。

#### 阶段2：业务优先级重排序

**核心原理**：将「BM25算法得分」与「业务优先级分数」线性叠加，重新排序后返回topK：

```Plain Text

最终得分 = BM25得分 + 优先级加分（国标+3000/园区规则+2000/历史案例+1000）
```

- **设计优势**：

    1. 业务逻辑与检索算法分离：ChemicalRAG只负责业务优先级，KnowledgeBaseService只负责算法，符合「单一职责原则（SRP）」；

    2. 保证合规优先级：即使“历史案例”的BM25得分更高，国标文档也会因加分优先返回（符合化工合规审核的业务规则）；

    3. 粗召回+重排序：既保证算法的召回率，又保证业务的合规性。

系统设计严格遵循企业级架构原则，核心模式/原则的应用如下：

### 4.1 核心设计模式

|模式/原则|应用场景|技术价值|
|---|---|---|
|策略模式|`IKnowledgeBaseService`接口|检索算法可替换（BM25/向量检索/TF-IDF），只需实现新的接口实现类，无需修改业务层。|

### 4.2 耦合度设计的核心逻辑

耦合度评分（1-10，1最低）体现了「核心模块低耦合、非核心模块可接受高耦合」的设计思路：

- **低耦合核心**：ChemicalRAG（耦合度2）、KnowledgeBaseService（耦合度3）—— 保证系统扩展能力；

- **可接受高耦合**：Program.cs（耦合度7）—— 入口层直接实例化所有服务，简单场景下无需过度设计，优化方向是引入DI容器（如Microsoft.Extensions.DependencyInjection）；
