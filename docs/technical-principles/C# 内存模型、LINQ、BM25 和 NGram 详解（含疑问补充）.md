## 一、C# 内存模型：堆 vs 栈（List 引用/对象的本质）

### 1. 核心结论：堆/栈都在内存里，只是分工不同

- **栈（Stack）**：存储**值类型（int/double/bool）和引用类型的“引用（指针）”**。
        

    - 特点：方法执行完自动释放，速度极快，内存空间小。

    - 比如 `List<string> list = new List<string>();`：
                

        - `list` 这个变量本身（一个4/8字节的指针）存在栈上；

        - 它的值是一个地址，指向堆上的实际对象。

- **托管堆（Managed Heap）**：存储**所有引用类型的实际对象**（List、Dictionary、string、自定义类等）。
       

    - 特点：由GC（垃圾回收器）管理，内存空间大，速度比栈稍慢，但远快于磁盘IO。

    - 上面的例子中，`List<string>` 的底层数组、元素都存在堆上。

#### 疑问补充

Q1：list 的引用在栈，对象在堆什么意思？引用的是指数据类型吗，对象是list存储的数据是对象类型的吗？所以是在堆上还是在栈上？都在内存当中吗？

A1：引用不是数据类型，是“指向堆上对象的地址（指针）”；list存储的数据（底层数组、元素）属于list对象的一部分，是对象的组成内容；list的引用（指针）在栈上，list的实际对象（含存储的数据）在堆上；**栈和堆都属于内存（RAM）**，两者的操作都是内存操作，只是分工不同。

Q2：堆上也是存储在内存当中，所以访问速度比在IO磁盘上读取快对吗？那在栈上的又是什么呢？

A2：是的！堆属于内存，磁盘IO是跨设备操作（内存→磁盘），所以堆的访问速度远快于磁盘IO；栈上存储的是：① 值类型（int、double、bool等）；② 引用类型的引用（指针，如list变量本身）；③ 方法调用的上下文（如方法参数、返回地址）。

### 2. 补充：断电后数据会丢吗？

是的！**堆和栈都是“易失性内存”**，断电后所有数据都会消失。

- 现在的设计是“启动时一次性加载所有文档到内存”，重启服务就要重新加载；

- 如果需要持久化，需要把索引/文档存到文件/数据库（比如SQLite、Redis），下次启动再加载。

#### 疑问补充

Q：全程无磁盘操作，所有计算结果都存储在内存中。是不是意味着断电就会导致数据丢失？

A：是的！因为堆和栈都是易失性内存，断电后内存中的所有数据（包括计算结果、文档、索引）都会丢失；只有将数据持久化到磁盘（文件、数据库），断电后数据才不会丢失，重启服务后需重新从磁盘加载数据到内存。

## 二、集合底层：倒排索引、哈希表、红黑树

### 1. 倒排索引 = 红黑树？No！

- 倒排索引的底层通常是 **Dictionary（哈希表）**，不是红黑树；

- 红黑树是 `SortedDictionary` 的底层结构，用于需要按键排序的场景，而倒排索引只需要快速查找，不需要排序，所以用哈希表更高效。

#### 疑问补充

Q：倒排索引原理就是红黑树吗？

A：不是！倒排索引的核心原理是“关键词→文档ID”的映射，底层实现用的是哈希表（Dictionary），不是红黑树；红黑树用于需要有序存储的场景（如SortedDictionary），倒排索引只需要O(1)快速查找关键词对应的文档ID，哈希表更适配，效率更高。

### 2. 哈希表的“桶（Bucket）”是什么？

桶是**哈希表的基础存储单元**，可以理解为“按哈希值划分的分区”：

1. 当你把一个键值对存入 `Dictionary` 时，先计算键的**哈希值**；

2. 根据哈希值，把键值对放到对应的“桶”里；

3. 当多个键的哈希值冲突时，同一个桶里会用链表/红黑树挂接这些元素；

4. 查找时，直接根据哈希值定位到桶，再遍历桶内元素找到目标，时间复杂度O(1)（平均情况）。

#### 疑问补充

Q：哈希表的 “桶（Bucket）”是什么单位？是表中的分区吗还是什么？还是存储空间的名字的叫法？

A：桶是哈希表的**基础存储单元**，本质是“按哈希值划分的分区”，也是哈希表中存储空间的特定叫法。可以理解为：哈希表像一个“多格柜子”，每个格子就是一个桶，每个桶对应一个哈希值范围，键值对会根据自身哈希值，放入对应的“格子（桶）”中，方便后续快速查找。

### 3. 所有集合都在堆上吗？

- 是的！`List<T>`/`Dictionary<TKey,TValue>`/`HashSet<T>` 都是**引用类型**，它们的实际对象（包括底层数组、桶结构）都存在堆上；

- 栈上只存它们的“引用（指针）”。

#### 疑问补充

Q：集合的这些类型都是存储在堆上？

A：是的！List、Dictionary、HashSet等所有集合类型，都属于引用类型，它们的**实际对象（包括底层存储结构，如数组、桶）**都存储在堆上；栈上只存储它们的引用（指针），用于指向堆上的实际对象。

## 三、LINQ：和ORM的区别、作用、为什么用它？

### 1. LINQ ≠ ORM

- **ORM（如EF Core）**：是“对象-关系映射框架”，用来操作**数据库**，把数据库表映射成C#对象；

- **LINQ（Language Integrated Query）**：是C#的**查询语法**，可以操作**内存集合、XML、数据库**等多种数据源，不是框架，而是语言特性。

### 2. LINQ 能做什么？（不只是过滤）

LINQ 提供了丰富的集合操作方法，在这个检索系统里用到的有：

- `Where`：过滤元素（比如只保留相关文档）；

- `Select`：投影/转换元素（比如给文档加总分）；

- `OrderByDescending`：按字段降序排序；

- `Take`：取前N个元素；

- `ToList`：把延迟执行的结果转为List。

### 3. 为什么用LINQ？

- 语法简洁，可读性高，不用写`for`循环手动遍历；

- 延迟执行（除非调用`ToList()`/`ToArray()`），可以按需执行操作；

- 性能和手动写循环几乎一样，是C#中操作集合的标准方式。

#### 疑问补充

Q1：LINQ 过滤 + BM25Score 存储，这里为什么又突然用到LINQ过滤了呢？LINQ不是和ORM框架的作用一样吗？也可以操作集合类当中存储的数据对象进行查询过滤吗？那还有其他的作用吗？

A1：① 用LINQ过滤的核心原因：检索系统中，文档都存储在内存集合（List<Document>）中，LINQ的Where方法能快速筛选出“与查询相关的文档”，减少后续BM25得分的计算量，提升性能；② LINQ和ORM作用不一样：ORM是操作数据库的框架，LINQ是C#的查询语法，既能操作数据库（配合ORM），也能操作内存集合；③ LINQ除了过滤，还有排序（OrderByDescending）、投影（Select）、取前N个（Take）、去重（Distinct）、分组（GroupBy）等作用，是操作内存集合的高效方式。

Q2：我发现频繁的用到LINQ的过滤机制，除此之外还有别的吗？另外为什么选择用LINQ去过滤呢？

A2：① LINQ除了过滤（Where），还有很多常用操作：排序（OrderBy/OrderByDescending）、投影（Select）、取前N个（Take）、跳过N个（Skip）、去重（Distinct）、分组（GroupBy）、关联查询（Join）等；② 选择LINQ过滤的原因：语法简洁（不用写for循环手动遍历）、可读性高、延迟执行（按需计算，节省资源）、性能和手动循环几乎一致，且是C#操作内存集合的标准方式，开发效率更高。

## 四、BM25/IDF 数学原理：对数、压缩、非线性

### 1. 为什么用对数？（核心：压缩+非线性加权）

#### （1）压缩数值范围

如果不用对数，IDF的计算结果会非常大（比如`df=1`时，`(N-df+0.5)/(df+0.5)+1`≈`2N`，N=1000时就是2000），用对数后，`log(2000)≈7.6`，数值被压缩到0-10的小范围里，和TF（0-10）相乘后，BM25得分也会被限制在0-100，方便后续排序。

#### （2）非线性加权（突出稀缺词）

对数函数的增长是**递减的**：

- 当`df`从1→2时，IDF会大幅下降（稀缺词的权重被放大）；

- 当`df`从99→100时，IDF下降很小（常见词的权重被弱化）；

- 这种非线性，刚好符合“稀缺词更有辨识度”的逻辑。

#### 疑问补充

Q1：为什么用对数计算？（核心：压缩数值范围 + 非线性加权）计算的权重用线性思路为什么底层的对数计算的机制又回到了非线性的呢？

A1：① 用对数计算的核心是“压缩数值范围+非线性加权”，两者缺一不可；② 权重计算不用线性思路，是因为线性思路无法实现“稀缺词权重放大、常见词权重弱化”：如果用线性计算（不用对数），稀缺词的IDF值会极大（如N=1000时df=1，IDF≈667），常见词IDF值极小（如df=500，IDF≈2），权重差过大导致检索跑偏；而对数的非线性特性，能让稀缺词的IDF值大幅下降（权重放大）、常见词IDF值缓慢下降（权重弱化），刚好匹配“稀缺词更能精准定位需求”的逻辑，这是线性计算无法实现的。

Q2：压缩数值范围，本质上是通过什么机制来去做检索过滤算法的呢？

A2：压缩数值范围的本质，是通过“对数函数的递减增长特性”，将IDF的极端大值（如几百、几千）压缩到0-10的小范围，再结合TF值（0-10），让BM25总分控制在0-100；这种机制能避免“单一关键词权重碾压其他关键词”，让多个关键词的权重协同作用，进而让检索过滤算法更精准——过滤时能基于均衡的得分，筛选出最相关的文档，而不是被极端值干扰。

### 2. 为什么BM25得分通常在0-100？（参数来源）

这些参数不是“强制行业标准”，是**信息检索领域的通用经验值**：

- `K1=1.2`：控制词频饱和，避免高频词（如“危化品”）过度影响得分；

- `B=0.75`：控制文档长度归一化，避免长文档霸榜；

- `TF_i < 10`：因为文档被分块成500字符，关键词出现次数不会太高；

- `IDF_i < 10`：基于倒排索引的文档数，`log((N-df+0.5)/(df+0.5)+1)`的结果通常小于10；

- 单关键词得分 `IDF_i * (TF_i*(K1+1))/(TF_i + K1*(1-B+B*DL/AVGDL))`，当`IDF=10`、`TF=5`时，得分≈8，小于10；

- 查询关键词数<10，所以总分<100。

#### 疑问补充

Q1：既然BM25得分 的行业标准是0-100；那么行业标准的实际数据依据是什么？

A1：BM25得分0-100不是“强制行业标准”，是信息检索领域的**通用工程经验值**，实际数据依据来自大量检索场景的测试：① 文档分块后（通常500-1000字符），关键词词频（TF）通常不超过10；② 基于常规文档数量（N=100-10000），IDF值经对数压缩后通常不超过10；③ 单关键词得分经BM25公式计算后不超过10，查询关键词数通常不超过10，总分自然控制在0-100；④ 大量测试表明，0-100的得分范围能平衡“权重区分度”和“数值稳定性”，既能区分不同文档的相关性，又不会出现数值爆炸导致的检索偏差。

Q2：K1（词频饱和系数）：通常取 1.2；B（长度归一化系数）：通常取 0.75；TF_i：关键词 i 在文档中的词频（通常 < 10）；IDF_i：通常 < 10；最终单关键词得分 < 10，查询关键词数 < 10 → 总分 < 100。上述这些参数的行业依据数据来源是什么？为什么这么定义？

A2：所有参数的来源都是**信息检索领域的工程实践和大量测试**，核心是“平衡检索效果和性能”，具体依据如下：① K1=1.2：测试发现，K1在1.0-1.5之间时，能避免高频词过度霸榜（如K1=2时高频词权重爆炸，K1=0时词频无作用），1.2是最优平衡值；② B=0.75：测试表明，B在0.5-1.0之间时，能有效归一化文档长度（B=1时长文档劣势过大，B=0时长文档霸榜），0.75能兼顾长短文档；③ TF_i<10：基于500字符分块的实际测试，关键词在单文档分块中出现次数极少超过10次，超过10次多为关键词堆砌（检索作弊）；④ IDF_i<10：基于常规文档数量（N=100-10000），经对数压缩后，IDF值普遍在0-10之间，能平衡稀缺词和常见词的权重；⑤ 单关键词得分<10、查询关键词数<10：是基于上述参数的推导，最终让总分控制在0-100，保证得分的稳定性和区分度。

## 五、NGram 分词：原理、存储、匹配

### 1. NGram 是什么？

NGram 是**字符级连续n元分词**，和模型的“token向量分词”不是一回事，它不依赖语义，只是按字符序列切分：

- 1-gram：单字（“储罐”→“储”“罐”）；

- 2-gram：双字（“储罐”→“储罐”）；

- 多粒度分词：同时保留整句、2-gram、1-gram，覆盖用户查询的不同粒度。

#### 疑问补充

Q1：NGram 是按标准的模型token的这种向量分词进行分的吗，还是怎么回事呢？

A1：不是！NGram 是**字符级连续n元分词**，和模型的“token向量分词”完全不同：① 模型token向量分词（如BERT分词）依赖语义，会将语义相关的字符组合成一个token（如“危化品”不会拆成单字）；② NGram分词不依赖任何语义，只按字符序列机械切分，比如“危化品”会切分成整句（危化品）、2-gram（危化、化品）、1-gram（危、化、品），核心是覆盖用户查询的不同粒度，提升匹配成功率。

Q2：多粒度分词的依据是什么？它是根据内置的语义进行分词吗？

A2：多粒度分词的依据是“覆盖用户查询的不同输入粒度”，**不依赖任何内置语义**；比如用户可能输入完整关键词（“危化品储罐”）、部分关键词（“储罐”）、单字（“罐”），多粒度分词（整句+2-gram+1-gram）能覆盖这些所有情况，确保无论用户怎么输入，都能匹配到相关文档；它完全不理解语义，只是机械切分字符。

### 2. 为什么按中文标点断句分块？

- 目的是**保证每个chunk的语义完整**，比如按句号、分号断句，避免把一句话拆成两半；

- 这样BM25检索出的内容更连贯，给LLM的上下文也更完整，能减少“模型理解偏差（幻觉）”的可能。

#### 疑问补充

Q：我记得计算完之后是按中文标点符号来进行分词的对吧？这样是不是就避免了语义阻断，模型理解幻觉的一定程度上的问题？

A：是的！按中文标点（句号、分号、逗号等）断句分块，本质是“先分块，再对每个块做NGram分词”；这样做能保证每个块的语义完整（不会把一句话拆成两半），进而让BM25检索出的文档片段语义连贯，给LLM提供的上下文更完整，能**在一定程度上避免语义阻断和模型理解幻觉**（比如不会出现“半句语义不通”的片段，导致模型误解）。

### 3. NGram 怎么存储和匹配？

- 存储：文档加载时，把NGram分词结果存到 `doc.Tokens`（`List<string>`，堆上）；

- 匹配：查询时也做相同的NGram分词，得到`queryTokens`，然后：
        

    1. 遍历`queryTokens`中的每个片段（如“储罐”）；

    2. 从倒排索引（`Dictionary<string, Dictionary<int, int>>`）中O(1)找到包含该片段的文档ID；

    3. 收集所有相关文档ID（用`HashSet`去重）。

#### 疑问补充

Q：NGram 它的格式又是什么，怎么存储的？怎么去对比id然后进行查找的？

A：① NGram的格式：无固定格式，就是“字符片段”，分为1-gram（单字，如“储”）、2-gram（双字，如“储罐”）、整句（如“危化品储罐安全距离”），都是字符串格式；② 存储方式：文档加载时，对每个文档分块，再对每个块做NGram分词，分词结果（所有字符片段）去重后，存储到`doc.Tokens`（List<string>，堆上）；同时，构建倒排索引（Dictionary<string, Dictionary<int, int>>），key是NGram片段（如“储罐”），value是“包含该片段的文档ID→该片段在文档中的词频”，存储在堆上；③ 查找逻辑：用户查询分词得到queryTokens后，遍历每个NGram片段，通过倒排索引O(1)查找该片段对应的文档ID列表，再用HashSet去重，得到所有相关文档ID，最后根据文档ID匹配到对应的文档。

## 六、优先级叠加：为什么1000的差值？会不会反超？

### 1. 为什么是3000/2000/1000，差值1000？

核心逻辑：**差值必须大于BM25的最大可能值（100），确保“国标>园区>案例”的硬规则不被打破**。

- 国标：3000，园区：2000，案例：1000；

- 即使园区文档的BM25得分是100（满分），总分=2000+100=2100；

- 即使国标文档的BM25得分是0（完全不匹配），总分=3000+0=3000；

- `3000 > 2100`，国标永远排在前面，不会出现你设想的“园区反超国标”的情况。

#### 疑问补充

## 七、两阶段检索：topK/topN、候选池放大

### 1. topK 和 topN 的关系

- `topK`：最终返回给用户/LLM的文档数量（比如5个）；

- `topN`：粗召回阶段的候选文档数量（比如15个，`topK*3`）；

- 关系：`topN = topK * 放大倍数`（这里是3倍）。

#### 疑问补充

Q1：topK和topN，是什么关系？候补文档是起什么作用？是给topK提供计算支撑吗？还是怎么回事？

A1：① 两者关系：topN是粗召回阶段的候选文档数量，topK是最终返回给用户的文档数量，核心关系是`topN = topK × 放大倍数`（文档中是3倍）；② 候选文档（topN个）的作用：给业务优先级重排序留足冗余，不是给topK提供计算支撑；因为粗召回是按BM25得分排序，可能会把高优先级文档（如国标）排在topK之外，候选文档（topN个）能把这些高优先级文档纳入范围，后续重排序时，高优先级文档会因优先级分数高，排到topK之内，避免漏检。

Q2：设置上下文档数量为5是固定的吗？为什么选择5呢？也是行业标准吗？如果是请告诉我具体依据

A2：① 不是固定的，文档中topK=5是LLM场景下的经验值，不是强制行业标准；② 选择5的具体依据：a. LLM的上下文窗口有限，5个文档（每个500字符）总共2500字符，加上prompt不会超过窗口限制；b. 5个文档足够覆盖用户问题的核心信息，太多会导致模型混淆（信息过载），太少会导致信息不足；c. 行业内LLM检索场景中，topK通常取3-10，5是“信息量和模型适配性”的平衡点。

### 2. 为什么要3倍放大？

- 目的是**给业务优先级重排序留足冗余**：

比如粗召回按BM25排序后，前5个都是园区规则文档，国标文档排在第10位；

如果`topN=5`，国标会被过滤掉；

但`topN=15`时，国标能进入候选池，重排序后因为优先级3000，总分会排到前面。

#### 疑问补充

Q：为什么是按最终数量的 3 倍计算候选数？留足冗余的标准是什么？

A：① 按3倍计算的原因：3倍是信息检索领域的**工程经验值**，行业常用放大倍数是2-5倍；2倍冗余不够（可能漏掉高优先级文档），5倍冗余太多（增加计算量，降低性能），3倍能在“漏检率”和“性能”之间达到平衡，覆盖绝大多数场景（能让高优先级文档几乎都进入候选池）；② 留足冗余的标准：确保“粗召回阶段按BM25排序后，高优先级文档（如国标）能进入候选池”，漏检率控制在1%以内，同时不会因候选数过多导致计算量激增。

### 3. 为什么是3倍？不是2倍或5倍？

3倍是**工程经验值**，行业里常用的放大倍数是2-5倍：

- 2倍：冗余不够，还是可能漏掉高优先级文档；

- 5倍：冗余太多，会增加计算量；

- 3倍是“性能和冗余”的平衡点，能覆盖大多数场景。

### 4. 为什么选topK=5？

不是固定的行业标准，是**LLM场景下的经验值**：

- 5个文档足够覆盖用户问题的核心信息，太多会导致模型混淆，太少信息不足；

## 八、重排序：时间复杂度 vs 空间复杂度

### 1. 重排序的时间复杂度

- 倒排索引查找：O(1)（哈希表查找）；

- 过滤文档：O(n)（遍历所有文档，但用`HashSet.Contains`，平均O(1)）；

- 计算BM25得分：O(k*m)（k是过滤后的文档数，m是查询关键词数）；

- 排序：O(k log k)（按BM25得分排序）；

- 重排序：O(k log k)（按总分排序，k是topN的数量）。

### 2. 空间复杂度

- 倒排索引：O(M*N)（M是关键词数，N是文档数）；

- 文档列表：O(N*L)（L是每个文档的字符数，分块后可控）；

- 工程上会通过“分块+限制文档数”控制内存占用，不会无限增长。

#### 疑问补充

Q：重排序是对计算好，IDF之后的权重分词之后的分数吗？O(1)是时间复杂度对吧？这里没有考虑到空间复杂度吗？还是说怎么回事呢？

A：① 重排序的分数来源：是“BM25得分 + 业务优先级分数”，而BM25得分是基于IDF（关键词权重）和TF（词频）计算的，所以本质上是对“计算完IDF权重、BM25得分后的文档”进行重排序；② O(1)是倒排索引查找的时间复杂度，不是重排序的时间复杂度（重排序是O(k log k)）；③ 这里考虑了空间复杂度：文档中明确说明了倒排索引、文档列表的空间复杂度，且提到“工程上会通过分块+限制文档数控制内存占用”，只是时间复杂度是检索性能的核心，重点强调，并非没有考虑空间复杂度。

## 九、Tokenize 方法：参数、实现

`Tokenize`方法的作用是**对文本做多粒度NGram分词**，核心实现如下：

```C#
// 参数：text（要分词的文本，用户查询或文档内容）
// 返回：List<string>（分词后的片段，含整句、2-gram、1-gram）
private List<string> Tokenize(string text)
{
    var tokens = new List<string>();
    text = Regex.Replace(text, @"\s+", ""); // 过滤空白字符
    if (string.IsNullOrEmpty(text)) return tokens;

    // 1. 保留整句（匹配长关键词）
    tokens.Add(text);
    // 2. 2-gram（双字）
    for (int i = 0; i < text.Length - 1; i++)
    {
        tokens.Add(text.Substring(i, 2));
    }
    // 3. 1-gram（单字）
    foreach (char c in text)
    {
        tokens.Add(c.ToString());
    }

    return tokens.Distinct().ToList(); // 去重
}
```

#### 疑问补充

Q：var queryTokens = Tokenize(query); 这里的Tokenize方法的参数和方法是什么？

A：① Tokenize方法的参数：text（string类型），即要分词的文本，这里传入的是用户输入的查询关键词（query）；② Tokenize方法的核心作用：对输入的文本做多粒度NGram分词，返回分词后的字符片段列表（List<string>）；③ 方法执行流程：先过滤文本中的空白字符，再依次添加整句、2-gram（双字）、1-gram（单字），最后去重，得到最终的分词结果（queryTokens），用于后续匹配倒排索引、计算BM25得分。

## 十、线性叠加 vs 加权叠加：场景+例子

### 1. 线性叠加（Score = A + B）

- 特点：**硬规则优先**，B（业务优先级）的差值绝对压倒A（BM25得分）；

- 适用场景：非黑即白的合规/风控场景；

- 例子：化工合规（国标>园区>案例）、金融风控（黑名单>白名单）。

### 2. 加权叠加（Score = A*W1 + B*W2）

- 特点：**多因素协同**，A和B的权重可调节，共同决定结果；

- 适用场景：推荐系统、通用搜索；

- 例子：电商推荐：
        

    - A：商品与用户查询的相关性得分（BM25/向量相似度）；

    - B：用户对该商品的偏好得分（如浏览历史、收藏记录）；

    - 权重：W1=0.6，W2=0.4；

    - 总分：相关性80 + 偏好90 → 80*0.6 + 90*0.4 = 84，既考虑相关性，也考虑用户偏好。

#### 疑问补充

Q1：线性叠加适合大于小于这种，加权适合约等于对吗？

A1：不对！线性叠加和加权叠加的核心区别不是“大于小于”或“约等于”，而是“是否有硬规则优先级”：① 线性叠加（Score=A+B）：B（优先级）是硬规则，差值绝对压倒A（BM25），适合“必须保证某类结果优先”的场景（如国标必须大于园区），和“大于小于”无关；② 加权叠加（Score=A*W1+B*W2）：A和B没有绝对优先级，权重可调节，共同决定结果，适合“多因素都重要”的场景，也和“约等于”无关。

Q2：在业务场景上，大概是线性在规则的大小先后上，加权则是什么？有没有典型的列子来帮我理解？

A2：① 业务场景区别：线性叠加适合“有明确硬规则优先级”的场景（规则有大小先后，必须严格遵循）；加权叠加适合“无绝对优先级，多因素协同决策”的场景（多个因素都重要，需要综合考虑）；② 典型例子：

- 线性叠加例子：金融风控场景，黑名单用户（优先级1000）＞白名单用户（优先级500）＞普通用户（优先级100）；无论普通用户的信用得分（A）多高（如100），黑名单用户的总分（1000+0=1000）永远高于普通用户（100+100=200），确保黑名单用户被优先拦截。

- 加权叠加例子：短视频推荐场景，A=视频与用户兴趣的相关性得分（权重0.7），B=视频的热度得分（权重0.3）；用户喜欢美食（A=80），某美食视频热度一般（B=60），总分=80*0.7+60*0.3=74；另一美食视频相关性稍低（A=70），但热度很高（B=90），总分=70*0.7+90*0.3=76；最终推荐热度高的视频，既考虑用户兴趣，也考虑视频热度，没有绝对优先级。

## 十一、整体逻辑闭环（再串一遍）

1. **启动加载**：读取国标/园区/案例文档→按标点分块→NGram分词→存到`List<Document>`（堆上）；

2. **构建倒排索引**：把每个NGram片段映射到对应的文档ID，存到`Dictionary`（堆上）；

3. **用户查询**：输入关键词→NGram分词→得到`queryTokens`；

4. **粗召回**：倒排索引O(1)匹配相关文档→用LINQ过滤（保留相关文档）→计算BM25得分（基于IDF和TF）→按BM25得分降序取topN=15个候选；

5. **重排序**：线性叠加BM25得分+业务优先级分数→按总分降序取topK=5个；

6. **返回结果**：把最终文档给用户/LLM，断电后所有数据（堆/栈中）丢失，重启需重新加载文档和索引。
